浏览器渲染原理

浏览器内核

常见的浏览器内核：

  • Trident（三叉戟）：IE、360安全浏览器、搜狗、百度、UC
  • Gecko（壁虎）：Mozila Firefox
  • Presto（急板乐曲）-> Blink：Opera
  • Webkit：Safari、360急速浏览器、搜高高速浏览器、移动端浏览器
  • Webkit -> Blink：Google Chrome、Edge

渲染流程

解析过程

HTML解析

默认情况下服务器会给浏览器返回index.html文件，解析HTML是所有步骤的开始。

浏览器在解析HTML的过程中构建DOM树。

生成CSS规则

在解析的过程中，如果遇到CSS的link元素，会由浏览器负责下载对应的CSS文件，下载CSS文件不会影响DOM的解析。

CSS文件下载完成后，会被解析为对应的规则树，称为CSSOM（CSS Object Model）。

构建渲染树

有了DOM Tree和CSSOM Tree后，可以结合构建Render Tree。

注意：

  • link元素不会阻塞DOM Tree的构建过程，但会阻碍Render Tree的构建过程，因为Render Tree在构建时需要对应的CSSOM Tree
  • Render Tree和DOM Tree并不是一一对应的关系，如display为none的元素不会出现在render tree中

布局和绘制

渲染树构建完成后，开始运行布局（Layout）以计算每个节点的几何体：

  • 渲染树会表示显示哪些节点以及其他样式，但不表示每个节点的尺寸、位置等信息
  • 布局是确定呈现树中所有节点的宽度、高度和位置信息

布局结束，将每各个节点绘制（Paint）到屏幕上。浏览器将布局阶段计算的每个frame转为屏幕上实际的像素点，包括将元素的可见部分进行绘制，比如文本、颜色、边框、阴影、替换元素等。

回流和重绘

回流

第一次确定节点的大小和位置称为布局，之后对节点的大小、位置修改重新计算称为回流（reflow）。

引起回流的情况：

  • DOM结构发生改变（添加或移除节点）
  • 改变布局
  • 修改窗口尺寸
  • 调用getComputedStyle方法获取尺寸、位置等信息

重绘

第一次渲染内容称为绘制，之后重新渲染称为重绘（repaint）。

引起重绘的情况：修改背景色、文字颜色等。

回流一定会引起重绘。

避免回流的情况：

  • 修改样式时尽量一次性修改
  • 尽量避免频繁操作DOM
  • 尽量避免使用getComputedStyle方法
  • 对某些元素使用position的absolute或fixed，这样不会对其他元素造成影响

composite合成

默认情况下，标准流中的内容都被绘制在同一个图层中，一些特殊的属性会创建一个新的合成层，每个合成层会单独渲染。分层可以提供性能，但会以内存管理为代价。

常见属性有：

  • 3D transforms
  • video、canvas、iframe
  • opacity动画转换时
  • position:fixed
  • will-change：一个实验性的属性，提前告诉浏览器可能发生哪些变化
  • animation或transition设置了opacity、transform

script元素和页面解析

script解析问题

浏览器在解析HTML过程中，遇到script元素会停止构建，下载JS代码并执行，执行结束后才会继续解析HTML。

原因是JS可以操作DOM，如果等DOM构建完成再执行JS代码，可能会造成严重的回流和重绘。但是如果JS代码处理时间较长会造成页面的解析阻塞。

为解决这个问题，script元素提供了defer和async属性。

defer

defer属性高柳浏览器不要等待脚本下载，继续解析HTML。如果脚本提前下载完成，会等待DOM树构建完成，在DOMContentLoaded事件之前执行defer中的代码。

注意：

  • 多个defer脚本可以保持正确的顺序执行
  • defer推荐放到head元素中
  • defer仅适用于外部脚本，对于script默认内容会被忽略

async

async于defer类似，它能让一个脚本完全独立：

  • 不能保证顺序，它独立下载、独立运行，不会等待其他脚本
  • 不会保证在DOMContentLoaded之前或之后执行

V8引擎

定义

V8是用C++编写的Google开源高性能JavaScript和WebAssembly引擎，它用于Chrome和Node.js等。

V8可以独立运行，也可以嵌入到任何C++应用程序中。

V8引擎的架构

  • Parse模块会将JS代码转为AST（抽象语法树），因为解释器不能直接解析JS代码。如果函数没有被调用，那就不会转为AST
  • Ignition是解释器，将AST转为ByteCode（字节码），同时收集TurboFan优化所需要的信息，如函数参数的类型信息，有类型才能进行真实的运算。如果函数只调用一次，Ignition会解释执行ByteCode
  • TurboFan是一个编译器，可以将字节码编译为CPU可以直接执行的机器码。如果一个被多次调用，就会被标记为热点函数，那么就会经过TurboFan转换成优化的机器码，提高代码的执行性能。机器码实际上也会被还原为ByteCode，因为如果后续执行函数的过程中，类型发生了变化，之前优化的机器码并不能正确地处理运算，就会逆向地转换成字节码

JavaScript执行原理

初始化全局对象

js引擎在执行代码之前，会在堆内存中创建一个全局对象：Global Object（GO）

  • 该对象所有的作用域都可以访问
  • 里面会包含Data、Array、String、Number、setTimeout、setInterval等
  • 其中有一个window属性指向自己

执行上下文

js引擎内部有一个执行上下文栈（Execution Context Stack，简称ECS），用于执行代码的调用栈。

它执行的是全局的代码块：

  • 全局的代码块为了执行会构建一个全局执行上下文（GEC）
  • GEC（Global Execution Context）会放入到ECS中执行

GEC放入到ECS中的内容包含两部分：

  • 在代码执行器，在parser转成AST的过程中，会将全局定义的变量、函数等加入到GlobalObject中，但是并不会赋值，这个过程称为变量的作用域提升
  • 在代码执行中，对变量赋值，或执行其他的函数

每个执行上下文会关联一个VO（Variable Object），变量和函数声明会被添加到这个VO对象中。

当全局代码被执行时，VO就是GO对象了。

函数执行过程

在执行的过程中执行到一个函数时，会根据函数体创建一个函数执行上下文（Functional Execution Context，简称FEC），并且压入EC Stack中。

每个执行上下文都会关联一个VO，函数执行上下文关联的VO称为AO（Activation Object）。

当进入一个函数执行上下文时，会创建一个AO对象，这个AO对象会使用arguments作为初始化，并且初始值时传入的参数，这个AO对象会作为执行上下文的VO来存放变量的初始化。

作用域和作用域链

当进行入到一个执行上下文时，执行上下文会关联一个作用域链（Scope Chain）。

作用域链是一个对象列表，用于变量标识符的求值。当进入一个执行上下文时，这个作用域链被创建，并且根据代码类型，添加一系列的对象。

作用域链在函数定义的时候已经确定，与调用的位置无关。

内存管理

认识内存管理

代码执行过程中需要分配内存，不管以什么样的方式来管理内存，内存的管理都会有如下生命周期：

  • 申请分配需要的内存
  • 使用分配的内存
  • 不需要使用时，对其进行释放

JS对于原始数据类型内存的分配会在执行时，直接在栈空间进行分配；

JS对于复杂数据类型内存的分配会在堆内存中开辟一块空间，并且将这块空间的指针返回值变量引用。

垃圾回收

因为内存的大小是有限的，当内存不再需要的时候，需要对其进行释放，以腾出更多的内存空间。

垃圾回收（Garbage Collection，简称GC），常见算法：

  • 引用计数：当一个对象有一个引用指向它时，这个对象的引用+1；当对象引用为0时就可以被销毁。弊端是循环引用的情况下无法回收
  • 标记清除：使用可达性分析算法，设置一个根对象，垃圾回收器定期从这个根开始，找所有从根开始有引用到的对象，对于没有引用到的对象就认为是不可用的对象
  • 标记整理：和标记-清除类似，不同的是，回收期间同时会将保留的存储对象搬运汇集到连续的内存空间，从而整合空闲空间，避免内存碎片化
  • 分代收集：对象分为新的和旧的两组，长期存活的对象，即旧的组被检查的频率降低
  • 增量收集：将垃圾收集工作分成几部分来做，然后将这几部分逐一处理，以避免大的延迟
  • 闲时收集：只在CPU空闲时尝试运行

闭包

定义

闭包在实现上是一个结构体，它存储了一个函数和一个关联的环境。闭包跟函数最大的区别在于捕捉闭包的时候，它的自由变量会在捕捉时被确定，这样即使脱离了捕捉时的上下文，它也能照常运行。

一个函数和对其周围状态的引用捆绑在一起，这样的组合就是闭包。

从广义的角度，JS中的函数都是闭包；

从狭义的角度，JS中的函数，如果访问了外层作用域的变量，那么它是一个闭包
