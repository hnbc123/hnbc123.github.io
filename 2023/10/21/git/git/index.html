Git基本使用

基本概念

版本控制

版本控制是一种软件工程技巧，借此能在软件开发的过程中，确保由不同人所编辑的同一程序文件都得到同步。

版本控制功能：

  • 不同版本的存储管理
  • 重大版本的备份维护
  • 恢复之前的项目版本
  • 记录项目的每一次修改
  • 多人开发的代码合并

集中式版本控制

CVS和SVN都是属于集中式版本控制系统，主要的特点是单一的集中管理的服务器保存所有文件的修订版本。协同开发人员通过客户端连接到这台服务器，取出最新的文件或提交更新。

集中式版本控制的核心问题是中央服务器不能出现故障。

分布式版本控制

Git属于分布式版本控制系统，客户端并不只提取最新版本的文件快照，而是把代码仓库完整地镜像下来，包括完整的历史记录。
任何一处协同工作的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复，因为每一次的克隆操作都是一次对代码仓库的完整备份。

Git本地仓库

Bash-CMD-GUI区别

  • Bash：Unix shell的一种，是Windows下的命令行工具，可以执行Linux命令。Git Bash是基于CMD的，在CMD基础上增添一些新的命令和功能。
  • Git CMD：命令行提示符，是WIndows操作系统上的命令行解释程序。
  • Git GUI：提供了一个图形用户界面来运行git命令。

Git的配置选项

安装Git后，第一件事要设置用户名和邮件地址。如果使用了–global选项，那么该命令只需运行一次。

获取Git仓库

有两种获取Git项目仓库的方式：
方式一：初始化git仓库，并且将当前项目文件添加到Git仓库中

方式二：从其他服务器克隆一个已存在的Git仓库

文件状态划分

需要对文件划分不同的状态，以确定这个文件是否已经归于Git仓库的管理：

  • 未跟踪：默认情况下，Git仓库下的文件没有添加到Git仓库管理中，需要通过add命令操作
  • 已跟踪：添加到Git仓库管理的文件处于已跟踪状态，Git可以对其进行跟踪管理

已跟踪文件可以进行细分状态划分：

  • staged：暂缓区中的文件状态
  • Unmodified：commit命令可以将staged中的文件提交到git仓库
  • Modified：修改了某个文件后，会处于Modified状态

可以使用 git status 命令查看文件状态

git忽略文件

一般总有些文件无需纳入git管理，也不希望它们出现在未跟踪文件列表，可以创建.gitignore文件。列出要忽略的文件模式。
在实际开发中，这个文件通常不需要手动创建，在必要时添加 自己的忽略内容即可。

git的校验和

Git中多有的数据在存储前都会计算校验和，然后以校验和来引用。
Git用以计算校验和的机制叫SHA-1散列，这是一个由40个十六进制字符组成的字符串，基于Git中文件的内容或目录结构计算出来。

查看提交历史

可以使用 git log 命令查看提交历史。
在不传入参数的情况下，git log会按时间先后顺序列出所有的提交。
这个命令会列出每个提交的SHA-1校验和、作者的名字和邮箱地址、提交时间以及提交说明。

git log –pretty=oneline –graph 命令可以使提交历史显示得更加清晰。

版本回退

如果想进行版本回退，需要先知道目前处于哪个版本：Git通过HEAD指针记录当前版本，HEAD总是指向该分支的最后一次提交。

如果想取消回退，可以使用 git reflog 命令查看所有日志，然后使用git reset回退。

Git 服务器仓库

远程仓库

在真实开发中，通常是多人开发的，所以会将管理的代码共享到远程仓库。
远程仓库(Remote Repository)通常是搭建在某一个服务器上的，所有需要在Git服务器上搭建一个远程仓库。
可以使用第三方Git服务器，比如GitHub、Gitee、Gitlab等，或者在自己服务器搭建一个Git服务。

远程仓库的验证

对私有的仓库进行操作，远程仓库会对我们的身份进行验证，主要有两种方式：

  • 基于HTTP的凭证存储
  • 基于SSH的密钥

HTTP协议是无状态的连接，所以每一个连接都需要用户名和密码。如果每次都这样操作会非常麻烦，所以Git拥有一个凭证系统来处理这个事情。

Secure Shell是一种加密的网络传输协议，可在不安全的网络中为网络服务提供安全的传输环境。
SSH以非对称加密实现身份验证，如果以SSH的方式访问Git仓库，就需要生成对应的公钥和私钥：

远程仓库的交互

  • 从远处仓库复制代码：git clone https://gitee.com/
  • 将代码推送到远程仓库：git push，默认将当前分支推送到origin远程仓库: git push origin master
  • 从远程仓库获取代码：git fetch，git fetch origin master
  • 合并本地仓库代码：git merge，git merge origin/master
  • 从远程仓库拉取代码：git pull，相当于git fetch + git merge

Git提交对象

几乎所有的版本控制系统都以某种形式支持分支。
在进行提交操作时，git会保存一个提交对象，该提交对象会包含一个指向暂存内容快照的指针；该提交对象还包含了作者的姓名和邮箱、提交时输入的信息以及指向它的父对象的指针。
首次提交产生的提交对象没有父对象，普通提交操作产生的提交对象有一个父对象，而由多个分支合并产生的提交对象有多个父对象。

Git分支

Git的分支本质上是指向提交对象的可变指针。Git的默认分支名为master，它会在每次提交时自动移动。

Git rebase

在Git中整合来自不同分支的修改主要有两种方法：merge和rebase。

rebase工作原理：

  • 首先找到两个分支的最近共同祖先C2
  • 对比当前分支相对于该祖先的历次提交，提取相应的修改并存为临时文件
  • 将当前分支指向目标基底C3
  • 以此将之前另存为临时文件的修改依序应用

rebase和merge的选择：

  • merge用于记录git的所有历史
  • rebase用于简化历史记录

永远不要在主分支使用rebase
