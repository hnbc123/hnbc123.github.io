axios请求方式

axios支持多种请求方式：

  • axios(config)
  • axios.request(config)
  • axios.get(url [, config])
  • axios.delete(url [, config])
  • axios.head(url [, config])
  • axios.post(url [, config])
  • axios.put(url [, config])

基本概念

版本控制

版本控制是一种软件工程技巧，借此能在软件开发的过程中，确保由不同人所编辑的同一程序文件都得到同步。

版本控制功能：

  • 不同版本的存储管理
  • 重大版本的备份维护
  • 恢复之前的项目版本
  • 记录项目的每一次修改
  • 多人开发的代码合并

集中式版本控制

CVS和SVN都是属于集中式版本控制系统，主要的特点是单一的集中管理的服务器保存所有文件的修订版本。协同开发人员通过客户端连接到这台服务器，取出最新的文件或提交更新。

集中式版本控制的核心问题是中央服务器不能出现故障。

分布式版本控制

Git属于分布式版本控制系统，客户端并不只提取最新版本的文件快照，而是把代码仓库完整地镜像下来，包括完整的历史记录。
任何一处协同工作的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复，因为每一次的克隆操作都是一次对代码仓库的完整备份。

Git本地仓库

Bash-CMD-GUI区别

  • Bash：Unix shell的一种，是Windows下的命令行工具，可以执行Linux命令。Git Bash是基于CMD的，在CMD基础上增添一些新的命令和功能。
  • Git CMD：命令行提示符，是WIndows操作系统上的命令行解释程序。
  • Git GUI：提供了一个图形用户界面来运行git命令。

Git的配置选项

安装Git后，第一件事要设置用户名和邮件地址。如果使用了–global选项，那么该命令只需运行一次。

获取Git仓库

有两种获取Git项目仓库的方式：
方式一：初始化git仓库，并且将当前项目文件添加到Git仓库中

方式二：从其他服务器克隆一个已存在的Git仓库

文件状态划分

需要对文件划分不同的状态，以确定这个文件是否已经归于Git仓库的管理：

  • 未跟踪：默认情况下，Git仓库下的文件没有添加到Git仓库管理中，需要通过add命令操作
  • 已跟踪：添加到Git仓库管理的文件处于已跟踪状态，Git可以对其进行跟踪管理

已跟踪文件可以进行细分状态划分：

  • staged：暂缓区中的文件状态
  • Unmodified：commit命令可以将staged中的文件提交到git仓库
  • Modified：修改了某个文件后，会处于Modified状态

可以使用 git status 命令查看文件状态

git忽略文件

一般总有些文件无需纳入git管理，也不希望它们出现在未跟踪文件列表，可以创建.gitignore文件。列出要忽略的文件模式。
在实际开发中，这个文件通常不需要手动创建，在必要时添加 自己的忽略内容即可。

git的校验和

Git中多有的数据在存储前都会计算校验和，然后以校验和来引用。
Git用以计算校验和的机制叫SHA-1散列，这是一个由40个十六进制字符组成的字符串，基于Git中文件的内容或目录结构计算出来。

查看提交历史

可以使用 git log 命令查看提交历史。
在不传入参数的情况下，git log会按时间先后顺序列出所有的提交。
这个命令会列出每个提交的SHA-1校验和、作者的名字和邮箱地址、提交时间以及提交说明。

git log –pretty=oneline –graph 命令可以使提交历史显示得更加清晰。

版本回退

如果想进行版本回退，需要先知道目前处于哪个版本：Git通过HEAD指针记录当前版本，HEAD总是指向该分支的最后一次提交。

如果想取消回退，可以使用 git reflog 命令查看所有日志，然后使用git reset回退。

Git 服务器仓库

远程仓库

在真实开发中，通常是多人开发的，所以会将管理的代码共享到远程仓库。
远程仓库(Remote Repository)通常是搭建在某一个服务器上的，所有需要在Git服务器上搭建一个远程仓库。
可以使用第三方Git服务器，比如GitHub、Gitee、Gitlab等，或者在自己服务器搭建一个Git服务。

远程仓库的验证

对私有的仓库进行操作，远程仓库会对我们的身份进行验证，主要有两种方式：

  • 基于HTTP的凭证存储
  • 基于SSH的密钥

HTTP协议是无状态的连接，所以每一个连接都需要用户名和密码。如果每次都这样操作会非常麻烦，所以Git拥有一个凭证系统来处理这个事情。

Secure Shell是一种加密的网络传输协议，可在不安全的网络中为网络服务提供安全的传输环境。
SSH以非对称加密实现身份验证，如果以SSH的方式访问Git仓库，就需要生成对应的公钥和私钥：

Git提交对象

几乎所有的版本控制系统都以某种形式支持分支。
在进行提交操作时，git会保存一个提交对象，该提交对象会包含一个指向暂存内容快照的指针；该提交对象还包含了作者的姓名和邮箱、提交时输入的信息以及指向它的父对象的指针。
首次提交产生的提交对象没有父对象，普通提交操作产生的提交对象有一个父对象，而由多个分支合并产生的提交对象有多个父对象。

Git分支

Git的分支本质上是指向提交对象的可变指针。Git的默认分支名为master，它会在每次提交时自动移动。

Git rebase

在Git中整合来自不同分支的修改主要有两种方法：merge和rebase。

rebase工作原理：

  • 首先找到两个分支的最近共同祖先C2
  • 对比当前分支相对于该祖先的历次提交，提取相应的修改并存为临时文件
  • 将当前分支指向目标基底C3
  • 以此将之前另存为临时文件的修改依序应用

rebase和merge的选择：

  • merge用于记录git的所有历史
  • rebase用于简化历史记录

永远不要在主分支使用rebase

概述

将需要处理的数据保存在应用程序的某个位置进行管理，称为状态管理。
在组件中定义data或在setup中返回使用的数据，称为state；
在template中可以使用这些数据，模块最终会被渲染成DOM，称为view；
在模块中会产生一些行为事件，处理这些行为事件时，有可能会修改state，这些行为事件称为actions。

Vuex将组件的内部状态抽离出来，以一个全局单例的方式管理。在这种模式下，组件树构成了一个巨大的View，不管在树的哪个位置，任何组件都能获取状态或触发行为。
Vuex通过定义和隔离状态管理中的各个概念，并通过强制性的规则来维护视图和状态间的独立性。

可使用npm install vuex进行安装。

默认情况下，模块内部的action和mutation仍然是注册在全局的命名空间中，这样多个模块能够对同一个action或mutation做出响应。

模块被注册后，它的所有getter、action及mutation都会自动根据模块注册的路径调整命名。

基本概念

webpack是一个为现代的JavaScript应用进行静态模块化打包的工具。它可以将代码打包成最终的静态资源部署到静态服务器。webpack默认支持各种模块化开发。

webpack中文官方文档：https://webpack.docschina.org/。

webpack的运行时依赖Node环境的，所有电脑上必须有node环境。

webpack的安装目前分为两个：webpack、webpack-cli。

执行webpack命令是依赖webpack-cli的，而webpack-cli中代码执行时，才是真正利用webpack进行编译和打包的过程。所以安装webpack时，需要同时安装webpack-cli。

path模块

path模块提供了一些处理路径和文件的方法，可以使用path模块解决不同系统路径分隔符的差异问题。

常见API：

  • 从路径获取信息
    • dirname：获取文件的父文件夹
    • basename：获取文件名
    • extname：获取文件扩展名
  • 路径的拼接：path.join
  • 拼接绝对路径：path.resolve
    • path.resolve()方法会把一个路径或路径片段的序列解析为一个绝对路径
    • 给定的路径的序列是从右往左被处理的，后面每个path被依次解析，直到构造完一个绝对路径
    • 如果在处理完所有给定path段之后，还没有生成绝对路径，则使用当前工作目录
    • 生成的路径被规范会并删除尾部斜杠，零长度path段被忽略
    • 如果没有path传递段，path.resolve()将返回当前工作目录的绝对路径

webpack配置文件

通常情况下，webpack需要打包的项目是非常复杂的，并且需要一系列的配置来满足要求，可以在根目录下创建webpack.config.js文件来作为webpack的配置文件。

如果配置文件名称不是webpack.config.js，可以通过–config来指定对应的配置文件。

webpack在处理应用程序时，会根据命令或配置文件找到入口文件，生成一个包含应用程序中所需的所有模块的依赖关系图，然后遍历图结构，打包一个个模块。

在webpack5之前，加载图片、文件等资源需要使用raw-loader、url-loader、file-loader等，在webpack5之后，可以使用资源模块类型来替代这些loader。

资源模块类型(asset module type)添加了4种新的模块类型：

  • asset/resource：发送一个单独的文件并导出URL
  • asset/inline：导出一个资源的data URI
  • asset/source：导出资源的源代码
  • asset：在导出data URI和发送一个单独的文件之间自动选择

自定义文件的输出路径和文件名的方式：

方式一：修改output，添加assetModuleFilename属性

方式二：在rule中添加generator属性，并设置filename

常用的placeholder：

  • [ext]：处理文件的扩展名
  • [name]：处理文件的名称
  • [hash]：文件的内容，使用MD4的散列函数处理

转换限制

在开发中往往把小的图片转为base64，和页面一起被请求，减少不必要的请求过程，而大的图片进行转换反而会影响页面的请求速度。

对图片转换大小进行限制的步骤：

  • 将type改为asset
  • 添加parser属性，并指定dataUrl的条件

resolve模块解析

模块解析规则

resolve用于设置模块如何被解析，它可以帮助webpack从每个require/import语句中，找到需要引入的合适的模块代码。

webpack能解析三种文件路径：

  • 绝对路径：不需要解析
  • 相对路径：使用import或require的资源文件所处的目录，会被认为是上下文目录。在import/require中给定的相对路径，会拼接此上下文路径来生成模块的绝对路径。
  • 模块路径：在resolve.modules中指定的所有目录检索模块，默认值是[‘node_modules’]

如果是文件：

  • 如果文件有扩展名，直接打包文件
  • 如果没有，将使用resolve.extensions选项作为文件扩展名解析

如果是文件夹：

  • 会根据resolve.mainFiles配置选项中指定的文件顺序查找，默认值是index
  • 再根据resolve.extensions来解析扩展名

extensions和alias配置

extensions是解析到文件时自动添加扩展名，默认值是[‘.wasm’, ‘.mjs’, ‘.js’, ‘.json’]

alias用于在项目目录结构比较深的时候，给某些常见路径起别名

npm

概述

npm，全称Node Package Manager，也就是Node包管理工具，目前在前端项目中也是用它来管理依赖的包。

npm属于node的一个管理工具，需要先安装Node。

npm的包是发布到registry上，当安装一个包时，其实是从registry上进行下载的。

配置文件

每个项目都会有一个对应的配置文件，这个配置文件记录项目的名称、版本号、项目描述等，也记录项目依赖的其他库的信息和依赖库的版本号。这个文件是package.json。

配置文件生成方式：

  • 手动创建项目，npm init -y
  • 通过脚手架创建项目，脚手架会生成package.json

常见的属性：

  • name：项目的名称，必填
  • version：当前项目版本号，必填
  • description：项目的基本描述
  • author：作者相关信息
  • license：开源协议
  • private：当值为true时，npm不能发布
  • main：设置程序的入口
  • scripts：用于配置一些脚本命令，以键值对的形式存在
  • dependencies：指定无论开发环境还是生成环境都需要依赖的包
  • devDependencies：在生成环境不需要的包会通过npm install webpack –save-dev安装到devDependencies属性中
  • peerDependencies：对等依赖，即依赖的包必须是以另外一个宿主包为前提。

依赖的版本管理

npm的包通常需要遵从semver版本规范，即X.Y.Z：

  • X主版本号：不兼容的API修改
  • Y次版本号：向下兼容的功能性新增
  • Z修订号：向下兼容的问题修正

版本号依赖下载：

  • x.y.z：明确的版本号
  • ^x.y.z：x保持不变，y和z安装最新的版本
  • ~x.y.z：x和y保持不变，z安装最新的版本

package-lock.json

package-lock.json文件解析：

  • name：项目的名称
  • version：项目的版本
  • lockfileVersion：lock文件的版本
  • requires：使用requires来跟踪模块的依赖关系
  • dependencies：项目的依赖
    • version：实际安装的版本
    • resolved：记录下载的地址，registry仓库中的位置
    • required/dependencies：记录当前模块的依赖
    • integrity：用来从缓存中获取索引，再通过索引去获取压缩包文件

npm常用命令

安装npm包分两种情况：

  • 全局安装：npm install webpack -g
  • 项目（局部）安装：npm install webpack

局部安装分为开发时依赖和生产时依赖：

  • 默认安装开发和生产依赖：npm install axios
  • 开发依赖：npm install webpack -D
  • 根据package.json中的依赖包：npm install

npm install会检测是否有package-lock.json文件。

如果没有：

  • 分析依赖关系。因为包可能会依赖其他的包，并且多个包之间会产生相同依赖。
  • 从registry仓库中下载压缩包
  • 获取到压缩包后会对压缩包进行缓存（从npm5开始）
  • 将压缩包解压到项目的node_modules文件夹中

如果有：

  • 检测lock中包的版本是否和package.json一致，若不一致重新构建依赖
  • 一致的情况下，优先查找缓存，若没找到从registry仓库下载
  • 找到缓存会获取压缩文件并解压到node_modules文件夹中

其他命令：

  • 卸载某个依赖包：
    • npm uninstall package
    • npm uninstall package –save-dev
    • npm uninstall package -D
  • 强制重新构建：npm rebuild
  • 清除缓存：npm cache clean

pnpm

pnpm可以理解为performant npm的缩写。

硬链接(hard link)：电脑文件系统中的多个文件平等地共享同一个文件存储单元。删除一个文件名字后，还可以用其他名字继续访问该文件。

符号链接(软链接soft link、Symbolic link)：是一类特殊的文件，其包含有一条以绝对路径或相对路径的形式指向其他文件或目录的引用。

当使用npm或Yarn时，如果有多个项目需要一个相同的依赖包，那么每个项目都会在硬盘中保存一份该依赖包的副本。

如果使用pnpm，依赖包将被存放在一个统一的位置，如果对同一依赖包使用相同的版本，那么磁盘上只有这个依赖包的一个文件；如果使用不同的版本，仅有版本之间不同的文件会被存储。

pnpm会创建非扁平的node_modules目录。

安装pnpm：npm install -g pnpm

npm命令 pnpm等价命令
npm install pnpm install
npm install pnpm add
npm uninstall pnpm remove
npm run pnpm

pnpm7.0之后，统一存储位置为：%LOCALAPPDATA%/pnpm/store

可以通过pnpm store path获取当前活跃的store目录。

pnpm store prune可以从store中删除当前未被引用的包来释放store的空间。

模块化概述

模块化开发的最终目的是将程序划分成一个个小的结构，在这个结构中编写自己的逻辑代码，有自己的作用域，定义变量名称时不会影响到其他结构。可以将指定的变量、函数、对象等导出给其他结构使用，也可以导入其他结构的内容。

JavaScript在ES6（2015）时才推出模块化方案，在此之前，为了让JavaScript支持模块化，有很多不同的模块化规范，如AMD、CMD、CommonJS等。

CommonJS是一个规范，Node对其进行了支持和实现，在Node中每一个js文件都是一个单独的模块，这个模块中包含CommonJS规范的核心变量：exports、module.exports、require，可以使用这些变量来进行模块化开发。

exports导出

exports是一个对象，可以在这个对象中添加很多属性，添加的属性会导出。

在另一个文件导入：const bar = require('./bar');

这意味着这个bar变量等于exports对象，也就是require找到对应的exports对象并将其赋值给bar变量。

在Node中导出时，经常使用module.exports。CommonJS中没有module.exports的概念，为了实现模块导出，Node中使用的是Module的类，每个模块都是Module的一个实例，也就是module。所以Node中真正用于导出的是module.exports，因为module才是导出的真正实现者。

exports也可以导出的原因是module对象的exports属性是exports对象的一个引用。

也就是module.exports = exports = require导入的变量

require导入

require是一个函数，可以引入一个文件（模块）中导出的对象。

require(x)

如果x是Node核心模块，如path、http，直接返回核心模块并停止查找

如果x是以./或../或/开头：

  • 将X当做一个文件在对应目录下查找，如果有后缀名，按照后缀名查找；没有按x、x.js、x.json、x.node的顺序查找
  • 没有找到对应的文件，将x作为一个目录，查找其中的index.js、index.json、index.node
  • 没有找到，报错：not found

如果没有路径且x不是核心模块，按当前路径及上层路径依次查找node_modules文件夹里的index.js文件。

模块的加载过程

模块在第一次引入时，模块中的js代码会被运行一次。

模块被多次引入时，会缓存，最终只加载一次。每个模块对象module都有一个loaded属性，为true表示已加载。

如果有循环引入，Node采用深度优先算法。

CommonJS加载模块是同步的，等到对应的模块加载完毕，当前模块中的内容才能被运行。

ES Module

概述

ES Module和CommonJS的模块化不同之处在于它使用了import和export关键字，采用编译期的静态分析，也加入了动态引入的方式。

export负责将模块内的内容导出；import负责从其他模块导入内容。

采用ES Module将自动采用严格模式：use strict

export关键字

export关键字将一个模块中的变量、函数、类导出。方式如下：

  • 在语句声明的前面加export关键字
  • 将所有需要导出的标识符放到export后面的{}中
  • 导出时通过as关键字给标识符起一个别名

import关键字

import关键字负责从另外一个模块中导入内容，有以下方式：

  • import {标识符列表} form ‘模块’
  • 导入时可通过as给标识符起别名
  • 通过*将模块功能放到一个模块功能对象上

export和import可以结合使用：

默认导出（default export）

默认导出不需要指定名字，在导入时不需要使用{}，并且可以自己指定名字。

在一个模块中，只能有一个默认导出。

import函数

通过import加载一个模块，是不可以将其放到逻辑代码中的，因为ES Module在被JS引擎解析时，就必须知道它的依赖关系。

如果希望动态加载某个模块，需要使用import()函数，import函数返回一个Promise，可以通过then获取结果。

ES Module的解析流程

ES Module的解析过程分以下三个阶段：

  • 构建：根据地址查找js文件，并且下载，将其解析成模块记录
  • 实例化：对模块记录进行实例化，并且分配内存空间，解析模块的导入和导出语句，把模块指向对应的内存地址
  • 运行：运行代码，计算值，并且将值填充到内存地址中

h函数

概述

如果在某些场景下需要JavaScript的完全编程能力，这时可以使用渲染函数，它比模板更接近编译器。

Vue在生成真实的DOM之前，会将节点转换成VNode，而VNode组合在一起形成一颗树结构，就是虚拟DOM（VDOM）。

h函数是一个用来创建vnode的函数，它接收三个参数：

  • tag：标签名，可以是HTML标签名或组件，必需
  • props：与attribute、prop和事件相对应的对象
  • children：子VNodes，使用h()构建

基本使用

h函数可以在两个地方使用：

  • render函数选项中
  • setup函数选项中

基本使用

Vue的transition动画

Vue提供了transition的封装组件，在以下情形中可以给任何元素和组件添加进入/离开过渡：

  • 条件渲染
  • 动态组件
  • 组件根节点

当插入或删除包含在transition组件中的元素时，Vue将会做如下处理：

  • 自动嗅探目标元素是否应用了CSS过渡或动画，如果有，在恰当的时机添加/删除CSS类名
  • 如果transition组件提供了JS钩子函数，这些钩子函数将在恰当的时机被调用
  • 如果没有找到JS钩子并且也没有检测到CSS过渡/动画，DOM插入、删除操作会立即执行

过渡动画class

  • v-enter-from：定义进入过渡的开始状态
  • v-enter-active：定义进入过渡生效时的状态
  • v-enter-to：定义进入过渡的结束状态
  • v-leave-from：定义离开过渡的开始状态
  • v-leave-active：定义离开过渡生效时的状态
  • v-leave-to：离开过渡的结束状态

class命名规则：
默认以v-作为前缀，如果transition有name属性，会以name的值作为前缀。

同时设置过渡和动画

如果同时使用过渡和动画，并且某一个动画执行结束时，另一个动画还没有结束，可以设置type属性为animation或transition来明确监听的类型。

也可以通过duration属性显示指定过渡时间。
duration可以设置两种类型的值：

  • number类型：同时设置进入和离开的过渡时间
  • object类型：分别设置进入和离开的过渡时间

过渡模式

如果不希望同时执行进入和离开动画，需要设置transition的过渡模式：

  • in-out：新元素先进行过渡，完成之后当前元素过渡离开
  • out-in：当前元素先进性过渡，完成之后新元素过渡进入

如果希望渲染一个列表，并且该列表中添加删除数据也希望有动画，要使用组件来完成。

  • 默认情况下，它不会渲染一个元素的包裹器，但可以指定一个元素并以tag attribute进行渲染
  • 过渡模式不可用
  • 内部元素需要提供唯一的key attribute
  • CSS过渡的类将会应用在元素内部，而不是这个组本身

在列表过渡的移动动画中，新增或删除有动画，但其他需要移动的节点没有动画。可以通过v-move来完成动画，它会在元素改变位置的过程中应用。
